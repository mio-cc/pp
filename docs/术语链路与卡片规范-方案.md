## 一、链路构成（现状全景）

### 1) 数据流水线（写入只走 CSV→build，主库只读）

```
data/raw/terms_seed.csv            ← 唯一写入入口（csv.DictReader 按“列名”读，列序不敏感）
        │  python scripts/build_kb.py
        ▼
build_kb.py
  ├ sync_volumes()    按 config/volumes.json 给每卷“播种”顶层分类行
  ├ ensure_category() 按 category 的 ' / ' 路径，递归建分类树节点
  ├ INSERT terms      写术语 + tags/term_tags + term_relations
  └ 导出              Markdown 页 / JSONL / web JSON / RAG chunks(term_chunks)
        ▼
SQLite 主库 visual_prompt_terms.sqlite（唯一权威源）
  volumes · categories · terms · tags · term_tags · term_relations · term_chunks
  + 全文索引 terms_fts(porter) + terms_fts_tri(trigram)
        ▼
API  api/app.py  serialize_term() 统一出参
        ▼
前端  useKB.js 拉数 → App.vue（侧栏分类树 / 术语网格 / 详情卡 / 提示词篮 PromptDock）
```

### 2) 体系 / 分类 / 术语的结构模型

```
卷（体系，V01–V15）                         ← 顶层，必须在 config/volumes.json 注册
└ 顶层分类（首段，须 ∈ 该卷 config 声明）
  └ 子分支 … 子分支（' / ' 分隔，深度不限，任意层级嵌套上下级包含）
      └ 原子术语（最小单位叶子）  ×N        ← 能直接当提示词的最具体取值
```

规则：一个节点**要么继续分支、要么挂原子术语**（不混层）；同一最深路径下的多个术语互为可选项（互斥靠**结构**表达，不靠字段；前端篮内并存标红）。

## 三、核心：把字段做成「模版 / 接口」（防跑偏的关键）

**为什么**：手写 CSV 是历史错误的温床——列序错位、逗号未转义、漏填必填、definition 复读名字/套话、顶层分类自造。改为「**JSON 模版 + Schema 接口 + 注入脚本**」三件套，AI 不再直接碰 CSV。

### 3.1 术语接口（草案）— `docs/templates/term.schema.json`

```json
{
  "$schema": "https://json-schema.org/draft/2020-12/schema",
  "title": "VisualPromptTerm",
  "type": "object",
  "additionalProperties": false,
  "required": ["term_uid","zh_term","en_term","volume_code","category",
               "definition_long","visual_effect","prompt_usage","use_cases","tags","status","version"],
  "properties": {
    "term_uid":        {"type":"string","pattern":"^V[0-9]{2}_T[0-9]{4}$","description":"卷内唯一递增"},
    "zh_term":         {"type":"string","minLength":1,"description":"中文名＝中文提示词，原子最小单位"},
    "en_term":         {"type":"string","minLength":1,"description":"英文名＝英文提示词"},
    "aliases":         {"type":"array","items":{"type":"string"},"default":[]},
    "volume_code":     {"type":"string","pattern":"^V[0-9]{2}$"},
    "category":        {"type":"string","pattern":"^[^/]+( / [^/]+)*$",
                        "description":"路径，' / '分隔；首段须∈该卷 config 顶层；末端为原子叶子"},
    "definition_long": {"type":"string","minLength":20,
                        "description":"真实详解(建议100-200字)；禁止复读名字、禁止占位/套话"},
    "visual_effect":   {"type":"string","minLength":4,"description":"画面上的视觉表现"},
    "prompt_usage":    {"type":"string","minLength":4,"description":"提示词里怎么用"},
    "use_cases":       {"type":"array","items":{"type":"string"},"minItems":1},
    "related_terms":   {"type":"array","items":{"type":"string"},"default":[]},
    "confused_with":   {"type":"array","items":{"type":"string"},"default":[]},
    "tags":            {"type":"array","items":{"type":"string"},"minItems":1},
    "source_refs":     {"type":"string","default":"整理"},
    "status":          {"enum":["draft","review","published","deprecated"]},
    "version":         {"type":"string","default":"V1.0"}
  }
}
```

JSON Schema 管不到的跨行约束，由注入脚本 / validate 兜底：term_uid 全局唯一且卷内递增、category 首段 ∈ config 该卷声明、同卷 zh_term 唯一、definition_long ≠ 复读 zh_term。

### 3.2 填充模版（草案）— `docs/templates/term.template.json`

```json
// 空白骨架（AI 复制后逐字段填；多值用数组，不用分号）
{
  "term_uid": "V01_T0025", "zh_term": "", "en_term": "",
  "aliases": [], "volume_code": "V01", "category": " / ",
  "definition_long": "", "visual_effect": "", "prompt_usage": "",
  "use_cases": [], "related_terms": [], "confused_with": [],
  "tags": [], "source_refs": "整理", "status": "published", "version": "V1.0"
}
```

```json

{
  "term_uid": "V01_T0024", "zh_term": "85mm人像", "en_term": "85mm Portrait Lens",
  "aliases": ["中长焦人像"], "volume_code": "V01",
  "category": "镜头与景深 / 焦距 / 定焦镜头",
  "definition_long": "85mm 定焦镜头透视压缩适中、背景虚化柔美，是人像摄影最常用焦段之一，能让主体从背景中干净分离。",
```

### 3.3 分类(链路)接口

- 顶层分类的“接口”＝ `config/volumes.json` 每卷 `categories` 数组（已对齐数据）。新卷必须先在此注册。
- 子分支无需预先声明——由 term 的 `category` 路径在 build 时自动建树。
- 可选增强：给每卷一个**树模版** `docs/templates/taxonomy.<Vxx>.json`（节点含 `children` 或 `terms`），脚本展开成 `category` 路径，帮 AI 先规划骨架再填叶子。

### 3.4 注入流程（闭环，AI 不直接碰 CSV）— 新增 `scripts/ingest_terms.py`

```
AI 产出 terms.json（term 对象数组，符合 term.schema.json）
   │  python scripts/ingest_terms.py terms.json
   ▼
① JSON Schema 校验：格式 / 必填 / 枚举 / 正则
② 跨行业务校验：uid 唯一递增、category 首段∈config、同卷 zh 唯一、definition_long 非复读非占位
③ 通过 → 数组转字段、自动转义、追加进 data/raw/terms_seed.csv（合并模式，可重跑）
④ 自动 build + validate；任一步失败则报错并**不写入**
```

这样把“列序/转义/漏填/套话/越级造分类”全部挡在写入前。

## 四、规则边界（明确版 · 针对历史错误逐条堵）

**✅ 必须**

| 规则 |
|---|
| 原子化：叶子是“能直接出图的最具体取值”（焦距到 85mm、颜色到深红色、色温到 3200K） |
| `category` 首段 ∈ 该卷 config 声明顶层；用 ' / ' 分隔；末端原子叶子 |
| 名字即提示词：`zh_term`/`en_term` 自身可直接当中/英文提示词 |
| `definition_long` 是真实详解，讲含义/原理/视觉/场景；建议 100–200 字 |
| 多值字段用 JSON 数组（经接口）；落到 CSV 由脚本转 `;` 并转义 |
| 12 个必填字段非空；`status∈{draft,review,published,deprecated}`，`version` 默认 V1.0 |

**⛔ 严禁**

| 禁止 | 纠正 |
|---|---|
| 笼统词当叶子（焦距/颜色/色温/镜头畸变） | 继续分支到具体值 |
| 概述词与子分类并列同层 | 概述词作分支名，叶子才是术语 |
| 顶层分类自造、不在 config 声明 | 先在 config 注册或并入已有顶层 |
| `definition_long` 复读名字 / 写“…是AI高频术语”等套话 / 占位 TODO | 必须写真实定义 |
| 重复 term_uid、同卷重复 zh_term | 注入脚本会直接拒绝 |
| 再填 `definition_short` / `positive_prompt` 等已删列 | 已不存在；名字即提示词 |
